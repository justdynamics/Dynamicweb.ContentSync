  <action>
    Per D-12: Add a "## Permissions" section to README.md, placed AFTER the "## Content Model" section and BEFORE the "## Configuration" section. The section should cover:

    1. **Overview paragraph**: ContentSync serializes and restores explicit page permissions. Pages using inherited permissions (no explicit overrides) are left unchanged during deserialization.

    2. **What gets serialized** sub-section:
       - Explicit page permissions only (not inherited)
       - Each permission entry includes: owner name, owner type (role or group), permission level
       - Permission levels: None, Read, Edit, Create, Delete, All
       - Example YAML snippet showing a page with Anonymous=None and AuthenticatedFrontend=Read

    3. **How permissions are restored** sub-section:
       - Roles (Anonymous, AuthenticatedFrontend, AuthenticatedBackend, Administrator) are matched by name -- role names are identical across all DW environments
       - User groups are matched by group name on the target environment (case-insensitive)
       - Existing explicit permissions on the target page are cleared first (source-wins model)
       - Pages without a permissions section in YAML are left untouched

    4. **Safety fallback** sub-section:
       - If a serialized group permission references a group that does not exist on the target environment, the group permission is skipped
       - As a safety measure, Anonymous access is set to None (deny) on that page to prevent accidental public exposure
       - The fallback is logged as a warning
       - Recommendation: ensure user groups exist on all target environments before deploying content with group-based permissions

    5. **Logging** sub-section (brief):
       - All permission actions are logged: applied, skipped (missing group), safety fallback triggered
       - Check scheduled task logs or API response for details
  </action>
