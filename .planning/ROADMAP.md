# Roadmap: Dynamicweb.ContentSync

## Overview

ContentSync delivers developer-controlled content synchronization for DynamicWeb through five phases that build strictly on each other: the shared data contract (DTOs + YAML) must exist before the predicate system, the predicate system must exist before the serialization pipeline, the serialization pipeline must produce files before deserialization can consume them, and the scheduled task wiring and AppStore packaging are thin integration work that come last. Each phase completes a coherent, independently verifiable capability before the next begins.

## Phases

## Phase Details

### Phase 1: Foundation
**Goal**: The shared data contract exists — plain DTO types, YAML round-trip fidelity proven, and mirror-tree file I/O working
**Depends on**: Nothing (first phase)
**Requirements**: SER-01, SER-02, SER-04
**Success Criteria** (what must be TRUE):
  1. All content node types (Area, Page, GridRow, GridColumn, Paragraph) have plain C# DTO types with no DynamicWeb dependencies
  2. A YAML file written by YamlDotNet round-trips without data loss for known-tricky strings: tilde (`~`), CRLF line endings, raw HTML, double quotes, and bang (`!`)
  3. FileSystemStore writes and reads a content tree to a mirror-tree folder layout where folder names reflect the content hierarchy and each item has exactly one `.yml` file
  4. Serializing the same content tree twice produces byte-for-byte identical YAML files (deterministic output)
**Plans:** 2/2 plans complete

### Phase 2: Configuration
**Goal**: A developer can define which content trees ContentSync operates on through a config file, and the predicate system correctly evaluates include/exclude rules
**Depends on**: Phase 1
**Requirements**: CFG-01, CFG-02
**Success Criteria** (what must be TRUE):
  1. A JSON config file on disk is loaded and validated at startup; missing or malformed config produces a clear error message
  2. A predicate rule with an include path matches items under that path and excludes items outside it
  3. An exclude rule overrides an include rule for overlapping paths
  4. The predicate can be evaluated in unit tests with no live DynamicWeb instance running

### Phase 3: Serialization
**Goal**: Running the serializer against a live DynamicWeb instance produces a complete, GUID-safe, deterministic YAML file tree on disk
**Depends on**: Phase 2
**Requirements**: SER-03, INF-02, INF-03
**Success Criteria** (what must be TRUE):
  1. Serializing the "Customer Center" content tree (pageid=8385) from instance A produces a YAML file tree where every item has a GUID identifier and no numeric cross-item references appear in any field
  2. The YAML files contain no data corruption: HTML content, multiline text, tildes, and CRLF sequences survive serialization unchanged
  3. File paths generated by the serializer do not exceed Windows MAX_PATH limits for the configured output directory; the serializer logs a warning and skips (not crashes) for any path that would overflow
  4. Serializing the same instance twice without content changes produces zero `git diff` output
**Plans**: TBD

### Phase 4: Deserialization
**Goal**: YAML files on disk can be applied to a target DynamicWeb instance with correct identity resolution, parent-before-child ordering, and atomic behavior
**Depends on**: Phase 3
**Requirements**: DES-01, DES-02, DES-03, DES-04
**Success Criteria** (what must be TRUE):
  1. Deserializing the "Customer Center" YAML files into a fresh instance B creates all pages, grid rows, and paragraphs with the correct content and hierarchy, matching the source instance A
  2. Items that already exist in instance B (matched by GUID) are updated in place; items not present in instance B are inserted with new numeric IDs assigned by DynamicWeb
  3. Deserialization never attempts to insert a child item before its parent exists in the database; the write order is always Areas first, then Pages, then GridRows, then Paragraphs
  4. If any write fails mid-run, the partially-applied state is either rolled back or clearly reported; the target database is never silently left in a broken partial state
  5. Dry-run mode reports exactly which items would be created, updated, or skipped without writing anything to the database
**Plans**: TBD

### Phase 5: Integration
**Goal**: ContentSync is a runnable DynamicWeb AppStore app — serialize and deserialize scheduled tasks appear in the DW admin, execute the full pipeline, and the package is distributable via NuGet
**Depends on**: Phase 4
**Requirements**: OPS-01, OPS-02, OPS-03, INF-01
**Success Criteria** (what must be TRUE):
  1. Two scheduled tasks appear in the DynamicWeb admin task list: one for serialization and one for deserialization; both can be triggered manually and run to completion
  2. Executing the serialize task from the DW scheduler produces the same YAML output as running the serialization pipeline directly in a test
  3. Every sync run logs a per-item structured summary: how many items were new, updated, skipped, and how many errors occurred, with each error including item GUID and context
  4. The NuGet package builds with the `dynamicweb-app-store` tag and can be added to a DynamicWeb project as a package reference
